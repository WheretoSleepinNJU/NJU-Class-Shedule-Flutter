# 上课鸭 Flutter 重构总架构与分块推进方案（基于 Class-Shedule-Flutter 现状审查）

## 1. 文档目标
- 给出从现有 `Class-Shedule-Flutter` 到新 UI/新交互的可执行迁移路线。
- 说明当前工程已具备能力、缺口、技术债与风险控制点。
- 明确每个阶段的交付物、开发要求、测试要求和验收标准。
- 单独回答“教务系统导入课表是如何实现的”。

## 2. 审查范围与证据来源
- Flutter 入口与状态：`lib/main.dart`、`lib/Utils/States/*`
- 核心页面：`lib/Pages/CourseTable`、`lib/Pages/Import`、`lib/Pages/Settings`、`lib/Pages/Share`
- 数据层：`lib/Models/*`、`lib/Models/Db/DbHelper.dart`
- 导入配置与脚本：`api/schoolList.json`、`api/tools/*.js`
- 工程能力：`pubspec.yaml`、`README.md`、`test/*`

## 3. 现有工程能力盘点

### 3.1 已完成能力（可复用）
1. 多平台基础：Android / iOS / OHOS，且具备主题、国际化、统计、更新检查能力。
2. 本地数据层：SQLite（课程表、课程）+ SharedPreferences（配置状态）。
3. 课程展示逻辑：
- 周次切换、课程冲突显示、非本周课程灰态、自由时间课程。
- 课程详情、删除、手动添加。
4. 课表导入能力：
- 内置导入（旧教务账号密码+验证码）。
- WebView 自动导入（统一认证/选课系统/在线学校列表）。
- 支持多学校在线配置与解析脚本远程下发。
5. 课表导出/导入能力：
- 二维码/分享串导入导出。
- payload 压缩、分片、校验和（CRC32）机制。
6. 设置能力：
- 主题、显示项、背景图、系统日历导出、Widget 配置。

### 3.2 关键缺口（与当前新 PRD 对比）
1. 缺失“待办域”完整实现：
- 当前工程无独立 Todo 数据模型、仓储、页面、统计链路。
2. 缺失“我的页”新结构与新弹窗体系：
- 现有设置页是传统列表，不是你已确认的新“我的主页面+卡片统计+二级弹窗”。
3. 缺失统一弹窗标准：
- 旧代码里大量 `showDialog` 分散，交互关闭规则、样式、文案策略不统一。
4. 状态管理可维护性不足：
- 现为 `scoped_model + SharedPreferences + 页面 Presenter` 混合模式，业务扩展复杂。
5. 领域层与 UI 层耦合偏高：
- 导入解析、数据写库、提示逻辑在页面/Presenter 里，后续扩展风险高。

## 4. 教务系统导入课表机制（你关心的核心问题）

### 4.1 实现方式总览
当前工程不是通过官方开放 API，而是“WebView 登录 + JS 注入解析 + 标准课程 JSON 落库”。

完整链路：
1. 入口页读取在线学校配置（`/schoolList.json`）。
2. 打开目标学校登录页（`initialUrl`），用户在 WebView 内完成认证。
3. 导航监听到目标地址（`targetUrl`）后，触发导入。
4. 根据平台下载解析脚本（`extractJSfileAndroid/iOS/OHOS`）。
5. 在 WebView 上下文执行脚本，抓取页面数据并转换为统一 JSON。
6. Flutter 侧反解字符串并 `json.decode`，得到学期名和课程列表。
7. 创建 `CourseTable`，遍历课程写入 SQLite `Course` 表。
8. 切换当前课表、刷新 UI，导入成功。

### 4.2 关键配置结构（schoolList.json）
每个学校配置至少包含：
- `initialUrl`：登录起点。
- `redirectUrl`：可选重定向标记。
- `targetUrl`：触发解析的页面。
- `preExtractJS`：解析前执行（比如展开课表页签）。
- `delayTime`：等待页面渲染秒数。
- `extractJSfile*`：按平台拉取解析脚本。
- `class_time_list`、`semester_start_monday`：可选学校时间模板。

### 4.3 解析脚本约定
脚本（如 `api/tools/njubksjw2.js`）在页面里执行，返回 encodeURIComponent(JSON)：
- 顶层：`{ name, courses }`
- 课程字段：`name/classroom/class_number/teacher/test_time/test_location/weeks/week_time/start_time/time_count/import_type/info...`

### 4.4 容错与失败处理
- 导入失败会记录错误码，并尝试把页面信息上报日志接口（`/log/log`）。
- 给用户弹错误对话框，支持复制错误码入群反馈。

### 4.5 结论
这套机制本质是“配置驱动的网页解析导入引擎”。
优势：适配学校扩展成本低；
风险：依赖学校页面结构，容易因页面改版失效。

## 5. 目标架构（重构后的推荐分层）

### 5.1 架构原则
1. 先稳定迁移，不做一次性推倒重来。
2. 领域分层清晰：UI / Application / Domain / Infrastructure。
3. 把“课程域”和“待办域”拆成独立 Feature。
4. 所有弹窗走统一 Modal 框架与规范。

### 5.2 建议目录（增量改造）
- `lib/app/`：AppShell、路由、主题、全局组件。
- `lib/features/schedule/`：课程表域。
- `lib/features/todo/`：待办域（新增）。
- `lib/features/profile/`：我的域（新增）。
- `lib/features/import/`：导入域（从旧页面剥离逻辑）。
- `lib/shared/`：通用组件、设计 token、错误处理、埋点。
- `lib/data/`：数据库、远程配置、解析脚本执行器、Repository 实现。

### 5.3 数据边界建议
1. 课程数据继续沿用 SQLite，先不换库。
2. 待办数据新增 `Todo` 表与统计快照表。
3. 用户配置先保留 SharedPreferences，后续逐步封装为 SettingsRepository。
4. 导入结果统一进入 `ImportResult`，再映射到领域模型，禁止页面直接操作原始 JSON。

## 6. 分块推进计划（必须按阶段验收）

## 阶段 P0：基线冻结与重构脚手架
### 目标
- 建立可并行开发的工程骨架，避免边开发边失控。

### 交付物
- 新目录骨架。
- 统一路由表。
- 设计 Token 文件（颜色/字体/间距/圆角/阴影）。
- Modal 基础组件（支持遮罩双击关闭）。

### 验收标准
- 新旧页面可共存运行。
- 任意页面可接入统一 Modal。
- CI 能通过基础检查与单测。

## 阶段 P1：课程域适配新 UI（保功能不减）
### 目标
- 先把课程表主页面/激活态/加号菜单迁移到新视觉。

### 交付物
- 新版课程表主界面。
- 课程激活态弹窗。
- 课程添加入口（自动导入/手动添加）。

### 验收标准
- 课程展示逻辑与旧版一致（周次、冲突、自由时间、灰态）。
- 新版交互流程与 PRD 一致。

## 阶段 P2：待办域从 0 到 1（核心新增）
### 目标
- 落地待办主页面、侧边栏、状态切换、统计回写。

### 交付物
- Todo 数据模型与仓储。
- 未完成/已完成列表页面。
- 新建待办流程与任务类型/课程类型弹窗。

### 验收标准
- 新建、编辑、完成、撤销、删除全链路可用。
- 与“我的页完成待办统计”实时联动。

## 阶段 P3：我的域与二级设置页
### 目标
- 完成“我的主页面 + 今日状态/已上课程/已完成待办弹窗”。

### 交付物
- 我的首页统计卡。
- 关于上课鸭、外观与主题、提醒与通知。
- 版本更新弹窗、提醒时间弹窗（课前提醒/每日滚轮）。

### 验收标准
- 统计口径正确。
- 所有弹窗支持关闭键与双击空白关闭。
- 二级页返回与数据回写正确。

## 阶段 P4：导入域重构（现有能力升级）
### 目标
- 保留原有学校导入能力并提高稳定性。

### 交付物
- `ImportEngine`（导航监听、脚本下载、执行、结果验证）。
- 学校配置缓存与降级策略（在线失败回退本地）。
- 导入错误分类与可追踪日志。

### 验收标准
- 当前已支持学校导入成功率不低于旧版。
- 页面结构变更时能快速热修脚本。

## 阶段 P5：分享导入导出与跨端一致性
### 目标
- 统一二维码/分享串规范，支持新待办扩展。

### 交付物
- payload 版本升级（例如 v2：含课程+待办）。
- 导入冲突处理策略（覆盖/合并/新建课表）。

### 验收标准
- 老版本二维码可兼容导入。
- 新版本支持数据校验和错误提示。

## 阶段 P6：质量与性能治理
### 目标
- 把“可用”提升到“可长期维护”。

### 交付物
- 单元测试：解析、仓储、统计口径。
- Widget/LiveActivity 关键链路测试。
- 崩溃与埋点监控看板。

### 验收标准
- 关键路径崩溃率达标。
- 核心用例自动化通过率 >= 95%。

## 阶段 P7：发布与灰度
### 目标
- 分平台灰度上线，降低导入与数据迁移风险。

### 交付物
- Feature Flag 开关。
- 数据迁移脚本。
- 回滚方案（版本回退 + 数据备份恢复）。

### 验收标准
- 灰度期间核心问题可 24h 内回滚。
- 用户数据零丢失。

## 7. 开发规范（执行层）
1. 每个 Feature 至少包含：`ui`、`application`、`domain`、`data`。
2. 禁止页面直接读写 SQLite，必须通过 Repository。
3. 导入脚本执行结果必须做 schema 校验。
4. 所有弹窗统一走 Modal 基类，统一关闭行为。
5. 埋点命名统一：`feature_action_result`（例如 `import_parse_success`）。

## 8. 测试与验收矩阵

### 8.1 功能验收
- 课程：导入、展示、冲突、切周、删除、分享。
- 待办：新建、完成、撤销、过滤、统计。
- 我的：统计卡、弹窗、设置项回写。

### 8.2 异常验收
- 网络断开时导入提示与重试。
- 解析失败错误码上报。
- 数据库迁移失败回滚。

### 8.3 视觉验收
- 按你当前 `操作逻辑说明和PRD` 全量文档逐项核对。
- 弹窗统一双击空白关闭规则必测。

## 9. 风险与应对
1. 学校页面结构变更导致导入失败：
- 预案：脚本 CDN 热修 + 本地缓存回退 + 错误码采样。
2. 旧状态管理与新模块冲突：
- 预案：按 Feature 渐进替换，保留桥接层。
3. 新增待办造成统计口径争议：
- 预案：先冻结统计定义，再开发。

## 10. 里程碑建议（可按两周一个 Sprint）
1. Sprint 1：P0 + P1
2. Sprint 2：P2
3. Sprint 3：P3 + P4
4. Sprint 4：P5 + P6 + P7

## 11. 最终交付定义（DoD）
- 所有你已确认的 UI/PRD 页面在 Flutter 端可复现。
- 课程与待办主链路稳定可用。
- 导入能力不低于旧版，且日志可追踪。
- 核心交互与设计标准全部有自动化或清单验收记录。
